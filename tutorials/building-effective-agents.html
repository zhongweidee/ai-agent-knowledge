构建有效的智能体（Building Effective Agents）

教程 Anthropic 2024.12

作者: Erik Schluntz, Barry Zhang
原文: Anthropic 官方

在过去一年中，我们与数十个团队合作，跨行业构建了大语言模型（LLM）智能体。始终如一地，最成功的实现并没有使用复杂的框架或专门的库。相反，他们使用简单、可组合的模式进行构建。

在这篇文章中，我们分享从客户合作和自己构建智能体中获得的经验，并为开发者提供构建有效智能体的实用建议。

什么是智能体？

"智能体"可以有多种定义方式。一些客户将智能体定义为完全自主的系统，可以长时间独立运行，使用各种工具完成复杂任务。另一些客户则用这个词来描述更确定性的实现，即遵循预定义的工作流。在 Anthropic，我们将所有这些变体归类为智能体系统（Agentic Systems），但在架构上区分了两种重要类型：工作流（Workflows）和智能体（Agents）。

类型定义
工作流（Workflows）LLM 和工具通过预定义的代码路径进行编排的系统
智能体（Agents）LLM 动态地指导自身流程和工具使用的系统，自主控制如何完成任务

何时（以及何时不）使用智能体

在构建 LLM 应用时，我们建议找到最简单的可行方案，只在必要时增加复杂度。这意味着可能根本不需要构建智能体系统。智能体系统通常以延迟和成本换取更好的任务性能，你需要权衡这种取舍是否合理。

当需要更高复杂度时：

  • 工作流为定义明确的任务提供可预测性和一致性
  • 智能体适用于需要灵活性和模型驱动决策的场景
  • 但对许多应用来说，用检索和上下文示例优化单次 LLM 调用通常就足够了

何时以及如何使用框架

有许多框架可以简化智能体系统的实现，包括：

  • LangChain 的 LangGraph
  • Amazon Bedrock 的 AI Agent 框架
  • Rivet，一个拖拽式 GUI LLM 工作流构建工具
  • Vellum，另一个构建和测试复杂工作流的 GUI 工具

这些框架通过简化底层任务（如调用 LLM、定义和解析工具、链式调用）让你快速上手。然而，它们通常会创建额外的抽象层，掩盖底层的提示和响应，使调试变得更困难。它们也容易让你在简单方案就足够时添加不必要的复杂性。

我们建议开发者直接使用 LLM API 开始：很多模式只需几行代码就能实现。如果确实要使用框架，请确保你理解底层代码。对内部机制的错误假设是客户错误的常见来源。

构建模块、工作流和智能体

在这一部分，我们将探讨在生产中看到的智能体系统的常见模式。从基础构建模块——增强型 LLM 开始，逐步增加复杂度，从简单的组合式工作流到自主智能体。

构建模块：增强型 LLM

智能体系统的基本构建模块是经过检索、工具和记忆增强的 LLM。我们当前的模型可以主动使用这些能力——生成自己的搜索查询、选择合适的工具、决定保留哪些信息。

我们建议关注实现的两个关键方面：

  1. 根据你的具体用例定制这些能力
  2. 确保为你的 LLM 提供简单、文档完善的接口

虽然有很多方式可以实现这些增强，一个方法是使用我们最近发布的 Model Context Protocol（MCP），它允许开发者通过简单的客户端实现与不断增长的第三方工具生态系统集成。

工作流：Prompt Chaining（提示链）

提示链将任务分解为一系列步骤，每个 LLM 调用处理前一个调用的输出。你可以在任何中间步骤添加程序化检查（如"门控"），以确保流程仍在正轨上。

何时使用：适用于可以清晰分解为固定子任务的任务。主要目标是以延迟换取更高的准确性，使每个 LLM 调用处理更简单的任务。

示例：

  • 生成营销文案，然后翻译成另一种语言
  • 撰写文档大纲，检查大纲满足特定标准，然后基于大纲撰写文档

工作流：Routing（路由）

路由将输入分类并导向专门的后续任务。这种工作流实现了关注点分离，可以构建更专业的提示。没有这种工作流，优化一种输入可能会损害其他输入的性能。

何时使用：适用于存在不同类别、需要分别处理的复杂任务，且分类可以准确完成（通过 LLM 或传统的分类模型/算法）。

示例：

  • 将不同类型的客服查询（一般问题、退款请求、技术支持）导向不同的下游流程、提示和工具
  • 将简单/常见问题路由到更小、更经济的模型（如 Claude Haiku），将困难/异常问题路由到能力更强的模型（如 Claude Sonnet），以优化性能和成本

工作流：Parallelization（并行化）

LLM 可以同时处理一个任务，并将它们的输出程序化地聚合。这种工作流有两种关键变体：

  • 分区（Sectioning）：将任务分解为独立子任务并行运行
  • 投票（Voting）：多次运行同一任务以获得多样化的输出

何时使用：当子任务可以并行化以加快速度，或者需要多个视角/尝试以获得更高置信度的结果时有效。

示例：

分区：

  • 实现护栏：一个模型实例处理用户查询，另一个筛选不当内容或请求
  • 自动化评估 LLM 性能，每个 LLM 调用评估模型性能的不同方面

投票：

  • 审查代码漏洞，多个不同的提示审查并标记代码中的问题
  • 评估给定内容是否不当，多个提示评估不同方面或要求不同的投票阈值

工作流：Orchestrator-Workers（编排器-工作者）

在编排器-工作者工作流中，一个中央 LLM 动态地分解任务，将它们委派给工作者 LLM，并综合它们的结果。

何时使用：适用于无法预知所需子任务的复杂任务（例如编码中，需要更改的文件数量和每个文件中更改的性质取决于具体任务）。与并行化的关键区别在于其灵活性——子任务不是预定义的，而是由编排器根据特定输入确定的。

示例：

  • 每次需要对多个文件进行复杂更改的编码产品
  • 涉及从多个来源收集和分析信息的搜索任务

工作流：Evaluator-Optimizer（评估器-优化器）

在评估器-优化器工作流中，一个 LLM 调用生成响应，另一个在循环中提供评估和反馈。

何时使用：当我们有明确的评估标准，且迭代改进能提供可衡量的价值时特别有效。两个适用标志：第一，当人类给出反馈时，LLM 响应可以明显改进；第二，LLM 能够提供这种反馈。

示例：

  • 文学翻译：翻译 LLM 可能最初无法捕捉细微差别，但评估 LLM 可以提供有用的批评
  • 复杂的搜索任务：需要多轮搜索和分析才能收集全面信息

智能体（Agents）

随着 LLM 在关键能力上的成熟——理解复杂输入、进行推理和规划、可靠使用工具、从错误中恢复——智能体正在生产中崭露头角。智能体从人类用户的指令或互动讨论开始工作。一旦任务明确，智能体就自主规划和运行，可能需要返回人类获取更多信息或判断。在执行过程中，智能体在每个步骤从环境获取"地面真相"（如工具调用结果或代码执行）以评估进度至关重要。然后智能体可以在检查点或遇到障碍时暂停以获取人类反馈。任务通常在完成后终止，但也常包含停止条件（如最大迭代次数）以保持控制。

智能体可以处理复杂的任务，但它们的实现通常很直接。它们通常只是在循环中使用工具、基于环境反馈的 LLM。因此，精心设计和清晰记录工具集至关重要。

何时使用：适用于开放式问题，难以或无法预测所需步骤数，且无法硬编码固定路径的场景。LLM 可能会运行多轮，你必须对其决策有一定程度的信任。智能体的自主性使其非常适合在可信环境中扩展任务。

智能体的自主性意味着更高的成本和潜在的复合错误。我们建议在沙盒环境中进行广泛测试，并设置适当的护栏。

示例：

  • 编码智能体解决 SWE-bench 任务（基于任务描述编辑多个文件）
  • 我们的"计算机使用"参考实现（Claude 使用计算机完成任务）

组合和定制这些模式

这些构建模块不是规定性的。它们是开发者可以根据不同用例塑造和组合的常见模式。成功的关键，与任何 LLM 功能一样，是衡量性能并迭代实现。再说一次：只有能明显地改进结果时，才应该增加复杂度。

总结

在 LLM 领域的成功不在于构建最复杂的系统，而在于为你的需求构建正确的系统。从简单的提示开始，通过全面的评估进行优化，只有在更简单的方案不足时才引入多步骤智能体系统。

在实现智能体时，我们遵循三个核心原则：

  1. 保持简单——在设计上保持简洁
  2. 优先透明——显式展示智能体的规划步骤
  3. 精心设计 ACI——通过全面的工具文档和测试来优化智能体-计算机接口（Agent-Computer Interface）

框架可以帮助你快速起步，但在进入生产环境时，不要犹豫减少抽象层，用基础组件构建。遵循这些原则，你可以创建不仅强大，而且可靠、可维护并受用户信任的智能体。

附录 1：实践中的智能体

A. 客户支持

客户支持将熟悉的聊天机器人界面与通过工具集成实现的增强能力相结合。这是更开放式智能体的天然应用场景，因为：

  • 支持交互自然遵循对话流程，同时需要访问外部信息和操作
  • 工具可以集成来拉取客户数据、订单历史记录和知识库文章
  • 退款发放、工单更新等操作可以程序化处理
  • 成功可以通过用户定义的解决方案清晰衡量

B. 编码智能体

软件开发领域在 LLM 功能方面显示出显著潜力，能力从代码补全演进到自主问题解决。智能体特别有效，因为：

  • 代码解决方案可以通过自动化测试验证
  • 智能体可以使用测试结果作为反馈迭代解决方案
  • 问题空间定义明确、结构清晰
  • 输出质量可以客观衡量

附录 2：提示工程你的工具

无论你构建什么类型的智能体系统，工具都将是重要组成部分。工具定义和规范应该像整体提示一样得到同等程度的提示工程关注。

关于工具格式的关键建议：

  • 给模型足够的 token 来"思考"，避免把自己逼入角落
  • 保持格式接近模型在互联网文本中自然见过的形式
  • 确保没有格式"开销"，比如需要精确计数数千行代码

经验法则：想想人机交互界面（HCI）投入了多少精力，并计划投入同样多的精力来创建良好的智能体-计算机接口（ACI）。

实用技巧：

  • 站在模型的角度思考：基于描述，使用这个工具是否显而易见？
  • 修改参数名称或描述，使事情更直观——就像为初级开发者写一份优秀的文档注释
  • 通过运行大量示例输入来测试模型如何使用你的工具，并迭代改进
  • 对你的工具进行防错设计（Poka-yoke）——修改参数使其更难以出错

例如，在构建 SWE-bench 的智能体时，我们发现模型在使用相对文件路径的工具时会出错——当智能体移出根目录后。为了解决这个问题，我们将工具改为始终要求绝对文件路径——然后模型完美地使用了这个方法。
